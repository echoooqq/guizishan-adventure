一 / 项目背景

为什么做这个项目

熟悉的场景让设计更具体
选择华师校园作为游戏世界，并非偶然。桂中路两旁种着桂花树，于是"桂花密语"谜题自然成立；图书馆是读书的地方，所以设计问答挑战；食堂里找饭卡是每个学生的日常，就成了搜索类谜题的灵感。场景本身就是设计灵感的来源，比凭空想象一个世界要高效得多，也更容易让设计逻辑自洽。
解谜类型的技术覆盖面
解谜游戏对系统设计的要求比其他类型更丰富——需要对话系统（条件分支、选项）、道具管理（拾取、组合、持久化）、状态判定（谜题解锁条件、昼夜触发）、数据驱动（对话/谜题/题库外置 JSON）等。一个项目就能覆盖多种编程范式。
全面体现课程所学
游戏是一个高度综合的软件系统，本项目涉及：面向对象设计（6 层类继承体系、33 个模块）、数据结构（状态机、碰撞矩阵）、文件 I/O（JSON 序列化存档）、模块化与解耦（依赖分层、接口通信），几乎把 Python 课的核心知识点都用上了。
从零搭建的工程挑战
Pygame 不是游戏引擎，而是底层渲染库——没有场景管理器、没有 UI 框架、没有动画系统。碰撞检测、状态机、摄像机、9-slice 边框、粒子系统，每一个都需要从底层实现。这种"造轮子"的过程，恰恰是对编程能力的综合锻炼。

游戏启动与状态机

你是一名刚踏入华师的新生。报到那天，校园里桂花逆季盛开，空气中弥漫着不寻常的气息——你误入了与现实校园重叠的秘境异世界。

10 种游戏状态
TITLE → 标题画面 INTRO → 开场动画 PLAYING → 游戏主状态 PAUSED → 暂停菜单 INVENTORY → 背包界面 MAP_VIEW → 全局地图 DIALOG → 对话框 PUZZLE → 谜题界面 QUEST_LOG → 任务日志 OUTRO → 结局动画
状态转换规则
每个状态有明确的进入/退出条件，不可跳转（如 PUZZLE 不能直接到 MAP_VIEW）。
PLAYING 是核心枢纽，所有子状态都从 PLAYING 进入并返回。
状态机保证了游戏流程的确定性和可维护性。

昼夜循环与秘境觉醒

白天，校园宁静如常；夜幕降临，秘境觉醒——幽绿光芒从地面涌出，桂花树在暗夜中发出微光，广场舞阿姨出现在博雅广场……

6:00 白天
18:00 黄昏
19:30 夜晚
6:00 次日
关键帧插值法
全天定义 11 个关键帧，每个帧指定色调颜色和 alpha 值。任意时刻的色调由相邻两个关键帧线性插值得到，实现平滑过渡。

白天 无叠加（秘境蛰伏态淡绿 alpha≈10）
黄昏 橘红→紫蓝渐变
夜晚 深蓝叠加 alpha 最高 155
觉醒态 幽绿脉冲 alpha 20-50 呼吸
NPC 时段系统
不同 NPC 在不同时段出现：图书管理员白天在岗，广场舞阿姨仅夜晚出现，桂中路谜题仅夜晚可解——昼夜不只是视觉变化，更影响游戏逻辑。
实现原理
# 11个关键帧定义色调 KEYFRAMES = [ (6.0, (0,0,0), 0), # 白天无叠加 (18.0, (200,100,30), 40), # 黄昏橘红 (19.5, (15,20,60), 155), # 夜晚深蓝 ... ] # 当前时刻插值 for i in range(len(KEYFRAMES)-1): t0, c0, a0 = KEYFRAMES[i] t1, c1, a1 = KEYFRAMES[i+1] if t0 <= hour <= t1: ratio = (hour-t0)/(t1-t0) color = lerp(c0, c1, ratio) alpha = lerp(a0, a1, ratio)
技术要点
所有叠加通过 Surface.set_alpha() 实现，每帧仅一次全屏 fill，性能友好。游戏内 1 天 = 现实 6 分钟，由 config.py 中 DAY_DURATION 控制。

夜晚限定机制
桂花密语仅在夜晚可触发——PuzzleManager 检查当前时钟时段，白天互动桂花树只返回普通对话。这是昼夜系统与谜题系统的跨模块联动。
密码输入终端
综合楼电脑终端支持数字键输入 4 位密码，含光标闪烁动画。线索散布在公告栏残页中，信息碎片化设计引导玩家主动探索。
搜索交互与对话驱动
食堂谜题没有独立 UI，完全通过对话系统+互动物体实现：搜索桌子→对话框反馈结果→找到饭卡→与阿姨对话归还。纯逻辑层 48 行代码，体现了数据驱动架构的简洁性。
谜题状态机
所有谜题共享 UNDISCOVERED → DISCOVERED → IN_PROGRESS → SOLVED 四阶段状态机，PuzzleManager 统一管理前置条件检查和徽章奖励发放。

题库 JSON 驱动

二 / 游戏设计
故事线 · 封印解除

数据驱动与存档系统

七枚徽章归位，喷泉涌出光芒，秘境封印解除。通关之后，回顾整个游戏的数据驱动架构——

数据驱动架构
所有面向玩家的内容存储在 JSON 数据文件中，代码只负责逻辑：
14 个对话 JSON 对话文本+条件分支+选项
items.json 道具名称/描述/分类/可组合性
puzzles.json 谜题名称/难度/前置条件/奖励
library_questions.json 问答题库

增删内容无需改代码，只需编辑 JSON 文件。
对话系统
逐字显示（50ms/字）+ 选项分支 + 条件显示（has_item、谜题状态等）+ 自动换行。支持游戏状态检查，同一 NPC 在不同进度下有不同对话。
存档安全 · 临时文件策略
采用"写临时文件 → 重命名"策略，防止写入中断导致数据损坏。存档涵盖 15+ 字段：玩家位置、背包、谜题状态、昼夜时钟、对话标记、探索迷雾、音频设置等，支持完整状态恢复。
自动存档触发
1 个自动存档 + 2 个手动存档槽。自动存档触发点：退出游戏、进/出室内、跨校区、解谜完成、拾取关键道具——关键节点不丢失进度。
JSON 异常捕获与降级
所有 JSON 加载均有 try-except，缺失字段使用合理默认值。地图加载失败时显示错误提示而非崩溃，保证健壮性。
三 / AI 辅助开发

大模型辅助开发流程

开发模式：AI 驱动 + 人工审查
1
需求描述 → 代码生成
用自然语言描述功能需求（如"实现昼夜循环，11个关键帧插值"），AI 生成初始代码框架
2
代码审查 → 修改迭代
人工审查生成代码的正确性、规范性和性能，提出修改意见，AI 迭代优化
3
运行测试 → Bug 修复
运行游戏测试功能，遇到 Bug 时将错误信息反馈给 AI，协作定位和修复
4
重构优化 → 系统集成
各模块完成后，AI 辅助模块间集成、依赖梳理和性能优化
AI 辅助的关键场景
美术资源生成
AI 生成角色头像(28×28)、7 枚徽章碎片图标、校园特色道具图标，再用 Piskel 手动微调统一风格
复杂算法实现
昼夜关键帧插值、9-slice 渲染、粒子系统、碰撞分轴检测等算法，AI 提供初始方案后人工调整参数
调试与问题定位
将运行时错误日志反馈给 AI，快速定位问题根因。如 pytmx 加载异常、中文字体渲染模糊、状态机转换死锁等
数据文件编写
14 个对话 JSON、道具/谜题数据、题库等大量结构化数据，AI 批量生成后人工校验内容准确性
